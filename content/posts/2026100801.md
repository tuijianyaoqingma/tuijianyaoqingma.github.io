## 大坑一：无限制的自动重试——“死循环”式烧钱

**问题所在**：生产环境中，网络波动、API限流、服务暂时超时是家常便饭。很多开发者为了追求“高可用”，会在代码里加上自动重试机制。结果呢？一旦某个环节出了问题，比如ERNIE API返回了一个速率限制错误，你的代码会像疯了一样地连续发送重试请求。

来看看这个“致命”的Node.js代码：

javascript
const MAX_RETRIES = 5;
async function askERNIERetry(prompt, retries = 0) {
  try {
    return await askERNIE(prompt);
  } catch (error) {
    if (retries < MAX_RETRIES) {
      console.log(`重试第${retries + 1}次...`);
      return askERNIERetry(prompt, retries + 1);
    } else {
      throw error;
    }
  }
}


看上去有最大重试次数（5次），安全了吧？**天真！** 

现实场景是：你的用户一次提交，可能触发了1000次并发调用。如果其中200次因为限流而失败，这200次就会触发5次重试，这意味着你瞬间多出1000次API调用。以ERNIE-4.0的Token消耗来算，一次中等长度的对话大约消耗500 Tokens。这1000次额外请求，就算每次只返回“请求过于频繁”，你的账单上也会平添 **500,000 Tokens** 的消耗。按照均价1元/万Tokens计算（基于[云雾api聚合站](https://www.yunwuai.cc/)的标准），这波操作直接烧掉你 **50块钱**。如果重试机制是递归的且没设置合理的指数退避，可能直接导致服务雪崩，同时烧光你的预算。

**正确做法**：使用指数退避（Exponential Backoff）和抖动（Jitter），并且在重试前确认“是否能重试”（比如429错误可以等，500错误则先暂缓）。

javascript
const sleep = (ms) => new Promise(resolve => setTimeout(resolve, ms));
async function askERNIESafe(prompt, retries = 3) {
  for (let i = 0; i < retries; i++) {
    try {
      return await askERNIE(prompt);
    } catch (error) {
      if (error.status === 429) {
        const waitTime = Math.min(1000 * Math.pow(2, i) + Math.random() * 1000, 10000);
        await sleep(waitTime);
        continue;
      }
      throw error; // 其他错误直接抛出，不重试
    }
  }
  throw new Error('重试耗尽');
}


**记住**：每一次不必要的重试，都是打在账单上的一颗子弹。

## 大坑二：无处不在的Token浪费——“富裕病”

**问题所在**：对于ERNIE这类任务型模型，很多人习惯在每次请求中都附带完整的提示词和很长的System Prompt，甚至把整个对话历史都一股脑塞进去。

来看一个常见的Node.js示例：

javascript
async function chatWithHistory(userMessage, chatHistory) {
  // chatHistory 可能包含最近20轮对话，有10000个Token
  const fullHistory = [
    {role: 'system', content: '你是一个AI助手，你需要回答用户的问题。你必须使用中文。你的回答要尽可能详细。你有能力处理复杂任务...'(这里可能写了200个Token)},
    ...chatHistory,
    {role: 'user', content: userMessage}
  ];

  const response = await client.chat.completions.create({
    model: 'ernie-4.0',
    messages: fullHistory,
    max_tokens: 2048 // 默认允许生成长响应
  });
  return response;
}


问题在于：
1.  **System Prompt过长且冗余**：很多人写System Prompt像写作文，200Toknes的废话，但核心意图50个Token就能说清。
2.  **对话历史全量上传**：用户只是问了个简单问题“今天天气怎么样？”，前面19轮讨论的“量子力学”和“如何制作蛋糕”的对话历史也被一起传进去了，白白浪费了 **10,000个Tokens** 用于处理这些无关历史。
3.  **贪婪的max_tokens**：对于“是/否”这类简单问题，设置`max_tokens: 2048`意味着模型会努力凑够字数，输出很多无关的说明文字，导致输出Token暴增。

**举个例子**：一个用户每天发500条消息，每条平均需要处理1000个Token。如果因为上述原因，每条消息多消耗了500个Token的无关历史，那么每天浪费的就是 **250,000个Tokens**。一个月下来就是 **7,500,000 Tokens**。在[云雾api聚合站](https://www.yunwuai.cc/)（www.yunwuai.cc）的1元/万Tokens定价下，这就是 **750块钱** 白白流走。

**正确做法**：精简System Prompt，使用滑动窗口策略只携带最近2~3轮对话，并根据任务类型动态调整`max_tokens`。

javascript
function buildMessages(userMessage, recentHistory = []) {
  const systemPrompt = '你是一个高效的问答助手。请基于用户问题，用最精炼的语言回答。';
  // 只取最近2轮对话
  const truncatedHistory = recentHistory.slice(-4); // 用户两轮 + 助手两轮
  return [
    { role: 'system', content: systemPrompt },
    ...truncatedHistory,
    { role: 'user', content: userMessage }
  ];
}

async function chatEfficiently(userMessage, recentHistory) {
  const messages = buildMessages(userMessage, recentHistory);
  const response = await client.chat.completions.create({
    model: 'ernie-4.0',
    messages: messages,
    max_tokens: 100 // 限制最大生成长度
  });
  return response;
}


**记住**：你的每一分钱都应该花在刀刃上，而不是花在运输历史垃圾信息上。

## 大坑三：单点故障的“倾巢之灾”——缺乏路由

**问题所在**：很多人死磕一个API提供方（比如直接用ERNIE的官网API），一旦官网限流、故障或者延迟过高，你的服务就直接挂了。更别说，这样做你完全没有价格和性能的优化空间。

假设你用`https://www.yunwuai.cc/v1`这样的API中转站，大部分时候没问题。但如果你的代码是“死写”的：

javascript
const API_BASE_URL = 'https://www.yunwuai.cc/v1'; // 写死，没有回退


这意味着，一旦[云雾api聚合站](https://www.yunwuai.cc/)的这个节点出现临时抖动，或者你的账户余额没了，你的ERNIE调用就会全部失败，导致用户体验极差。更糟的是，你无法利用其他模型做“降级”处理，比如当ERNIE-4.0太贵时，换用更便宜的ERNIE-Lite来处理一些简单的文本总结任务。

**正确做法**：实现一个简单的“路由策略”，备选多个模型和API地址，并设置优先级和熔断机制。

javascript
const providers = [
  { baseURL: 'https://www.yunwuai.cc/v1', model: 'ernie-4.0', priority: 1 },
  { baseURL: 'https://www.yunwuai.cc/v1', model: 'ernie-lite', priority: 2, costReduce: true },
  { baseURL: 'https://cloud.baidu.com/v2', model: 'ernie-4.0', priority: 3 } // 官方直连作为最后保底
];

async function robustAskERNIE(prompt, failedProviders = []) {
  for (const provider of providers) {
    if (failedProviders.includes(provider.baseURL + provider.model)) continue;
    try {
      const client = new openai({ apiKey: provider.apiKey, baseURL: provider.baseURL });
      const response = await client.chat.completions.create({
        model: provider.model,
        messages: [{ role: 'user', content: prompt }],
        max_tokens: 100
      });
      return response;
    } catch (error) {
      console.warn(`Provider ${provider.model} failed:`, error.message);
      failedProviders.push(provider.baseURL + provider.model);
      // 继续尝试下一个
    }
  }
  throw new Error('所有API提供方都失败了');
}


通过这种路由策略，你可以：
-   在高峰时段，自动将高消耗的ERNIE-4.0切换为更省钱的ERNIE-Lite，降低成本。
-   当主API地址不稳定时，自动切换到备用地址或官方接口。

**记住**：不要把所有鸡蛋放在一个篮子里。多一个选择，不仅多一份保障，也多一个省钱的空间。
