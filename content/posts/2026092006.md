## 代码防坑：常见错误与优化建议

你以为改了 `base_url` 就万事大吉？在实际调用中，有四个常见的坑，踩一个就可能让调试时间翻倍。

### 坑1：API Key格式错误

云雾API Key通常以 `sk-` 开头，但你复制时可能会多一个空格或少一个字符。**建议：直接用 `os.environ.get()` 从环境变量读取，避免硬编码。**

python
import os
api_key = os.environ.get("YUNWU_API_KEY")
if not api_key:
    raise ValueError("请设置环境变量 YUNWU_API_KEY")


### 坑2：模型名写错

虽然接口兼容OpenAI，但模型名必须用云雾平台支持的格式。Kimi K2 Thinking在云雾上的模型ID就是 `kimi-k2-thinking`。如果搞混，API会返回404或模型不存在错误。

**建议：写代码前，先在云雾后台的模型列表里确认一遍官方名称。**

### 坑3：忽略 `max_tokens` 与 `temperature` 的配合

Kimi K2 Thinking作为推理模型，深度思考时需要更大的输出长度。如果你设置的 `max_tokens` 过小（比如256），模型可能在推理中途就“哑火”了。

**优化建议：**

- 复杂推理任务：`max_tokens` 设定为 4096 或更高
- 创意写作任务：`temperature` 可设为 0.8 ~ 1.0
- 精确问答任务：`temperature` 建议 0.3 ~ 0.5

### 坑4：批量请求时没有加退避重试

高并发场景下，偶尔会遇到限流（429）错误。直接报错退出显然不行。

**优化建议：使用 `tenacity` 库加上指数退避重试。**

python
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, min=2, max=10))
def call_kimi_with_retry(client, messages, model="kimi-k2-thinking", **kwargs):
    response = client.chat.completions.create(
        model=model,
        messages=messages,
        **kwargs
    )
    return response

## 流式输出实战：让聊天体验起飞

大多数对话应用都离不开流式输出。下面是一个可直接粘贴的流式处理代码：

python
response = client.chat.completions.create(
    model="kimi-k2-thinking",
    messages=messages,
    stream=True
)

full_content = ""
for chunk in response:
    if chunk.choices[0].delta.content is not None:
        content = chunk.choices[0].delta.content
        full_content += content
        print(content, end="")  # 实时打印
    # 流式模式下，最后一个chunk会包含使用统计
    if chunk.choices[0].finish_reason == "stop":
        print("\n\n--- 流式输出完毕 ---")

# 获取Token使用情况（注意：流式模式下usage来自最后一个chunk）
try:
    total_tokens = response.response.usage.total_tokens  # 部分库需这样获取
except AttributeError:
    print("Token统计需要在非流式模式下获取更准确")
