### 聚合 SDK 的核心思路：把“选择”延迟到运行时

传统的模型接入方式，是把模型的 API 地址、密钥、模型名称全部硬编在代码里，或者在环境变量里配置多组 key，通过 switch-case 去选。这种做法的本质是 **“编译时”确定调用链**。

[云雾ai聚合平台](https://www.yunwuai.cc/)的聚合 SDK 颠覆了这个模式。它的工作原理很简单：你不需要在代码里指定“调用哪个模型”，而是在请求中只传递 `model` 参数，平台自动从你的授权模型中匹配最合适的那个。无论你写的是 `model = "deepseek-coder"` 还是 `model = "deepseek-v3"`，都走同一套 API 入口，`base_url` 统一设置为 `https://www.yunwuai.cc/v1`。

这样一来，代码里不再有“写死”的模型，所有决策延后到运行时。你甚至可以在 SDK 里动态切换模型，无需改一行业务逻辑。

### 不写死代码：用一次理解“动态API”的价值

为什么“不在代码里写死模型”是开发者的福音？因为绝大多数 AI 应用的生命周期里，模型调用策略是常变的。

- **预算调整**：月底要省钱，想从 GPT-4 切到 DeepSeek-R1 满血版，拿便宜模型跑大部分推理。
- **效果对比**：做 A/B 测试，临时换模型看哪个结果更符合场景需求。
- **故障切换**：某个模型 API 挂了，或官方涨价了，立刻切到备用模型。

传统方式下，每次切换都要改代码、改配置、重启服务。用聚合 SDK，你只需要在调用时改一个字符串变量。甚至可以在服务端写一个配置中心，让用户或运维人员动态修改 `model` 的值，实现零部署切换。

举个例子，你在代码里定义一个配置变量 `MODEL_NAME`：

python
# config.py
MODEL_NAME = "deepseek-coder"  # 线上改这个变量即可


然后整个项目只使用 `MODEL_NAME` 这个变量去调用 API。当你想切模型时，改这个文件，甚至通过环境变量或远程配置去控制，一切都归到 `https://www.yunwuai.cc/v1` 这一条管道里。

### 不配多环境：一套配置走天下

很多团队在做多模型接入时，会为每个模型维护一套环境变量文件：


# .env
OPENAI_API_KEY=xxx
OPENAI_BASE_URL=https://api.openai.com/v1
DEEPSEEK_API_KEY=yyy
DEEPSEEK_BASE_URL=https://api.deepseek.com/v1
CLAUDE_API_KEY=zzz
CLAUDE_BASE_URL=https://api.anthropic.com/v1


当项目需要在多个环境中跑（开发、测试、生产），这些环境变量配置就会变得极其混乱。每个环境都得手动确保所有 key 都正确，更新时还得同步。

使用[云雾ai聚合平台](https://www.yunwuai.cc/)的聚合 SDK 后，你只需要一套环境变量：


# .env (云雾聚合)
YUNWU_API_KEY=your-yunwu-key
YUNWU_BASE_URL=https://www.yunwuai.cc/v1


所有模型的调用都走这个入口。开发环境、测试环境、生产环境，只需要改这个 key（比如不同级别的 key 有不同的配额和权限），其他逻辑完全一样。

这种方式不仅减少了环境变量的数量，还降低了配置出错的风险。你在部署时再也无需检查“哪个模型的 key 忘了配了”，因为全平台就一个 key。
