## 第一个隐形消费：Token 计算黑洞

大多数 Node.js 示例代码长这样：

javascript

const response = await openai.chat.completions.create({

  model: 'gpt-5',

  messages: messages,

  max_tokens: 4096

});

看着没问题对吧？但问题就出在这个 `max_tokens: 4096`。企业级对话里，用户可能一口气问十几个问题，每次请求消耗的 Token 量根本不是固定值。你的代码每次调用都会从 prompt 里带走全部历史上下文，导致 Token 用量成倍增长，账单自然水涨船高。

**我的解法**：别用暴力截断，改用**滑动上下文窗口**。把历史消息精简成摘要，只保留最多 5 轮对话，剩下的用 `role: "system"` 压缩成一段指令。这样每次请求能省掉 30%-50% 的 Token 费用。

---

## 第二个隐形消费：无用上下文缓存

示例代码里常写的是：

javascript

const history = [];

history.push(userMessage);

history.push(assistantMessage);

把每轮对话完整保存，然后每次都塞进新请求里。结果就是，用户问一句无关痛痒的“今天天气怎么样”，整个对话历史都得跟着跑一遍。这不仅是算力浪费，更是现金流黑洞。

**正确做法**：
1. 只保留最近 3 轮有效对话。
2. 超出部分的旧对话，让模型自己做摘要，然后替换掉。
3. 把摘要放在 `system` 消息里，而不是用户消息里。

这样每次请求的体积就小了，Token 成本也跟着降。

---

## 第三个隐形消费：多轮对话的重复调用

很多实现方案是每次用户发消息都重新调用 API，哪怕只是问同一个问题的不同表述。结果就是，GPT-5 在处理“再解释一遍”这种请求时，又要从头算一遍。

**优化策略**：
- 实现请求去重机制（deduplication）。
- 对同主题的多次提问，先用缓存匹配历史答案。
- 只有确实没缓存或请求语义不同时才调用 API。

用 Node.js 实现也很简单，加一个内存哈希表或 Redis 缓存，命中率能到 40% 以上，直接省下近一半的 API 调用次数。

---

## 第四个隐形消费：无意义的重试逻辑

生产环境中，API 偶尔会超时或报错，于是示例代码里很贴心地写了：

javascript

for (let i = 0; i < 3; i++) {

  try {

    const res = await fetch(...)

    break;

  } catch (e) {

    // 重试

  }

}

问题在于，重试默认是 `await` 同步的，意味着每次失败都白白耗着算力和时间。更糟的是，重试时你大概率还是带着同样的 prompt 和 Token 去撞墙，不仅没解决问题，还白花了几次请求的钱。

**最佳实践**：
- 重试加上指数退避（Exponential Backoff），比如第一次等 2 秒，第二次等 4 秒。
- 重试前检查错误类型，只有网络错误才重试，业务错误（如模型拒绝）直接返回。
- 重试次数限制在 2 次以内，超过就降级返回预设结果。

这样既不浪费 Token，又保障了稳定性。

---

## 第五个隐形消费：错误信息泄露隐私

有些示例代码遇到错误就原样打印：

javascript

console.error('API Error:', error);

如果你在日志系统里记录了用户输入和错误细节，这些信息就可能变成隐私漏洞。更糟的是，有些开发者把 API Key 硬编码在代码里，一旦代码被分享，Key 就废了。

**安全措施**：
- 不要直接打印 error 对象，只打印错误码和类型。
- API Key 必须用环境变量加载，用 `dotenv` 或 `--env-file`。
- 日志里过滤掉用户身份信息，只保留匿名 ID。
