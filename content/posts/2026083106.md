### 坑3：忽略“上下文窗口”的计算方式

**血泪回忆：** 模型明明是 128k 的上下文，你怎么就报 `context_length_exceeded` 错误了呢？因为 Moonshot 的上下文 TPM 计算包含了输入的 prompt 和输出的 content 总和，不止是你看到的对话记录。

**避坑指南：**

在你每次创建巨量的 `messages` 列表之前，先加上计数器来控制轮次。比如你别一次性丢给他一万行Python代码，可以切分：

python
max_context_tokens = 100000  # 留点余量
# 然后循环计算当前所有消息的total_tokens
if total_tokens > max_context_tokens:
    # 弹出最早的一轮对话
    messages.pop(0)


这是一个极其关键的逻辑，否则跑着跑着就崩。

### 坑4：流式输出的坑——控制台一片空白

**血泪回忆：** 我用 `stream=True` 调用，结果控制台没有任何输出，我还以为没卡了。后来发现是忘了 `for` 循环读取 chunk。

**正确代码：**

python
stream = client.chat.completions.create(
    model="moonshot-v1-8k",
    messages=[{"role": "user", "content": "写一段200字的文案"}],
    stream=True
)

for chunk in stream:
    if chunk.choices[0].delta.content is not None:
        print(chunk.choices[0].delta.content, end='')


不写循环的话，流只会闪一下，你什么都没看到就结束了，非常坑。

**切片法：**

### 坑9：调用量的并发限制——接口返回429

**血泪回忆：** 我写了个 for 循环同时触发几个 API 调用来测试，直接就报了 `429 Too Many Requests`。

**避坑指南：**

2. 如果还是遇到，那你可以写一个简单的重试机制：

python
import time, random
def call_with_retry(client, model, messages, retries=3):
    for i in range(retries):
        try:
            return client.chat.completions.create(model=model, messages=messages)
        except Exception as e:
            if "429" in str(e):
                wait = 2**i + random.uniform(0, 1)
                time.sleep(wait)
            else:
                raise


这个函数可以帮你平稳规避限流。底层有一个典型的指数退避策略，能极大提高稳定性。
